This cookbook covers the most common use cases when using WLST-API.
Do not hesitate to suggest new recipes if you think they deserve to be included here.

As you should know by now, if you have carefully read the main WLST-API documentation, properties files, which are the main input for WLST-API scripts, can contain substitution variables.

Samples properties file included in this document contain some recurrent variables:

${domains.home}

Path to the domains directory where the domain directory will be created.
${weblogic.home}

This variable is provided by WLST-API and is the directory of your Weblogic installation, usually ${MIDDLEWARE_HOME/wl_server}

1. Creating Domains

Domain creation is handled by the scripts/domainCreatory.py script.

Usage:
	> bin/wlstapi.sh scripts/domainCreatory.py input.properties

1.1. Creating a basic domain

The following properties file creates a basic (with a minimum configuration) Weblogic 12c domain named BasicDomain listening on t3://localhost:7001.

Domain template location

The location for the domain template used to create the domain (and specified with property domain.template.location) has changed with the 12c release.

Weblogic 10, 11g

→ ${weblogic.home}/common/templates/domains/wls.jar
Weblogic 12c

→ ${weblogic.home}/common/templates/wls/wls.jar
domain-basic.properties
script.log.level = DEBUG
script.log.file = create_domain.log

################################
# Domain properties
domain.name = BasicDomain
domain.location = ${domains.home}/${domain.name}
domain.template.location = ${weblogic.home}/common/templates/wls/wls.jar
# pre-12c template: domain.template.location = ${weblogic.home}/common/templates/domains/wls.jar
domain.template.additional.scheme =
domain.production.mode = false
domain.jdbc.script.loader.dbversion = 10g

################################
# Admin Server properties
admin.server.name = AdminServer
admin.server.host = localhost
admin.server.port = 7001
admin.server.ssl.enable = false
admin.server.security.username = weblogic
admin.server.security.password = welcome1

1.3. Creating a clustered domain

This property file defines a domain with a two nodes cluster.

domain-cluster.properties
script.log.level = DEBUG
script.log.file = create_domain.log

################################
# Domain properties
domain.name = ClusterDomain
domain.location = ${domains.home}/${domain.name}
domain.template.location = ${weblogic.home}/common/templates/wls/wls.jar
# pre-12c template: domain.template.location = ${weblogic.home}/common/templates/domains/wls.jar
domain.template.additional.scheme =
domain.production.mode = false
domain.jdbc.script.loader.dbversion = 10g

################################
# Admin Server properties
admin.server.name = AdminServer
admin.server.host = localhost
admin.server.port = 7001
admin.server.ssl.enable = false
admin.server.security.username = weblogic
admin.server.security.password = welcome1

################################
# Configure machines
machine.items = 1
machine.0.name = LocalMachine
machine.0.type.unix = true

################################
# Configure servers
managed.server.items = 2

# Managed Server 1 and 2
managed.server.0.name = WlsServer1
managed.server.0.listen.address = ${admin.server.host}
managed.server.0.listen.port = 7010
managed.server.0.ssl.enable = false
managed.server.0.ssl.listen.port = 7011
managed.server.0.machine.name = ${machine.0.name}

managed.server.1.name = WlsServer2
managed.server.1.listen.address = ${admin.server.host}
managed.server.1.listen.port = 7020
managed.server.1.ssl.enable = false
managed.server.1.ssl.listen.port = 7021
managed.server.1.machine.name = ${machine.0.name}

#################################################
# Configure cluster
#################################################
cluster.items = 1

cluster.0.name = WlsCluster
cluster.0.managed.servers = ${managed.server.0.name},${managed.server.1.name}
cluster.0.messaging.mode = unicast
cluster.0.listen.address = ${admin.server.host}
It is also possible to create the cluster after a basic/regular domain has been created, in which case you would use the import script to create the cluster resources as you would do for any other Weblogic resource.

domain.production.mode = false

# allow t3s administration port
domain.administration.enable = true
domain.administration.mbean.auditing.enable = false
domain.administration.port = 9002
domain.administration.port.enable = true

################################

admin.server.security.username = weblogic
admin.server.security.password = welcome1

2. Connecting to a Weblogic domain

The online scripts (import.py, remove.py, export.py) require the definition of properties to specify how to connect to the Weblogic server. These properties are common for all online scripts and are usually placed at the top of the properties file.

Typical properties file header
script.log.level = DEBUG
script.log.file = wlstapi.log

################################
# Connection properties
domain.loading.type = connect            (1)
domain.connect.url = t3://localhost:7001 (2)
domain.connect.username = weblogic       (3)
domain.connect.password = welcome1       (4)

#...
1 script connection type, it should alway be "connect" for on-line scripts
2 admin server listen address, usually in the form of t3://hostname:port
3 user name
4 password

Note that it is also possible to authenticate (pass user name and password) through the command-line if you don’t want to supply these in the properties file (see main WLST-API documentation)

3. Creating resources

The import script import.py handles the creation of resources in a Weblogic domain.
Unless otherwise stated this script does not support the update of existing resources.

The input properties contains the list of resources to be created along with their attribute values.

This scripts must be run against a running server.

3.1. Domain

3.1.1. JTA Timeout

Starting from version 1.8, you can configure the global JTA timeout by including this property in your properties file:

...
# JTA Timeout in seconds (default is 30)
domain.jta.timeout = 300
...

3.1.2. Startup & Shutdown classes

You can configure Startup and/or Shutdown classes as illustrated below.

startup-shutdown.properties
script.log.level = DEBUG
script.log.file = test_import.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# StartupClass properties
domain.startup.class.items = 1
domain.startup.class.0.name = StartupClass-0
domain.startup.class.0.arguments = arg1 arg2
domain.startup.class.0.class.name = some.package.StartupClass
domain.startup.class.0.failure.is.fatal = false
domain.startup.class.0.load.after.apps.running = true
domain.startup.class.0.load.before.app.activation = false
domain.startup.class.0.load.before.app.deployments = true
domain.startup.class.0.targets = AdminServer

################################
# ShutdownClass properties
domain.shutdown.class.items = 1
domain.shutdown.class.0.name = ShutdownClass-0
domain.shutdown.class.0.arguments = arg1 arg2
domain.shutdown.class.0.class.name = some.package.ShutdownClass
domain.shutdown.class.0.targets = AdminServer

3.3. JMS Resources

3.3.1. Creating a Queue and a Topic

To create a JMS Queue or Topic, you first need to configure a persistent store, a JMS Server and a JMS module as well as a Connection Factory.
It is also recommended to create a subdeployment for the targeting of the JMS resources:

jms-queue-topic.properties
script.log.level = DEBUG
script.log.file = create_jms_resources.log

# Substitution variables
targetedServer = AdminServer
jmsModuleName = JMSModule

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Persistent store
persistent.filestore.items = 1
persistent.filestore.0.name = JMSFileStore
persistent.filestore.0.location = ${user.home}/temp/WeblogicFileStores/JmsQueueTopic
persistent.filestore.0.target = ${targetedServer}

################################
# JMS Server
jms.server.items = 1
jms.server.0.name = JMSServer
jms.server.0.store = ${persistent.filestore.0.name}
jms.server.0.target = ${targetedServer}

################################
# JMS Module
jms.module.items = 1
jms.module.0.name = ${jmsModuleName}
jms.module.0.targets = ${targetedServer}

################################
# JMS Subdeployment
jms.subdeployment.items = 1
jms.subdeployment.0.name = JMSSubDeployment
jms.subdeployment.0.module.name = ${jmsModuleName}
jms.subdeployment.0.targets = ${jms.server.0.name}

################################
# JMS ConnectionFactory
jms.connection.factory.items = 1
jms.connection.factory.0.name = ConnectionFactory
jms.connection.factory.0.jndi.name = jms/ConnectionFactory
jms.connection.factory.0.module.name = ${jmsModuleName}
jms.connection.factory.0.subdeployment.name = ${jms.subdeployment.0.name}
jms.connection.factory.0.xa.enable = true

################################
# JMS Queue
jms.queue.items = 1
jms.queue.0.name = JMSQueue1
jms.queue.0.jndi.name = jms/queue/JMSQueue1
jms.queue.0.module.name = ${jmsModuleName}
jms.queue.0.subdeployment.name = ${jms.subdeployment.0.name}

################################
# JMS Topic
jms.topic.items = 1
jms.topic.0.name = JMSTopic1
jms.topic.0.jndi.name = jms/topic/JMSTopic1
jms.topic.0.module.name = ${jmsModuleName}
jms.topic.0.subdeployment.name = ${jms.subdeployment.0.name}
Using a JDBC Store

If you wish to use a JDBC store instead of a File store, create a data source then replace the file store definition with the following one:

JDBC store
#...
jdbc.datasource.items = 1
jdbc.datasource.0.name = jdbcStoreDataSource
#...
persistent.jdbcstore.items = 1
persistent.jdbcstore.0.name = JdbcStore
persistent.jdbcstore.0.table.prefix = myJdbcStore_
persistent.jdbcstore.0.target = ${targetedServer}
persistent.jdbcstore.0.datasource = ${jdbc.datasource.0.name}
#...

3.3.2. Creating distributed destinations

The following script creates two Uniform Distributed Queues and one Uniform Distributed Topic.
The first queue configuration is basic while the second is exhaustive.
We assume that a cluster has been properly setup (see Creating a cluster).

jms-distributed.properties
script.log.level = DEBUG
script.log.file = create_jms_resources.log

#
# NOTE: We assume here that a cluster has been configured
#

# Substitution variables
cluster = Cluster1
server1 = ManagedServer1
server2 = ManagedServer2
adminServer = AdminServer
jmsModuleName = JMSModule

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Persistent stores
persistent.filestore.items = 2
persistent.filestore.0.name = JMSFileStore1
persistent.filestore.0.location = ${user.home}/temp/WeblogicFileStores/jms_cluster
persistent.filestore.0.target = ${server1}
persistent.filestore.1.name = JMSFileStore2
persistent.filestore.1.location = ${user.home}/temp/WeblogicFileStores/jms_cluster
persistent.filestore.1.target = ${server2}

################################
# JMS Servers

The following properties file will create a user and a group and will assign the user to the group:

sec-user-group.properties
script.log.level = DEBUG
script.log.file = usergroup.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Create user
security.users.items = 1
security.users.0.name = TestUser
security.users.0.password = welcome2
security.users.0.comment = This is a test user
security.users.0.realm = myrealm
security.users.0.authenticator = DefaultAuthenticator

################################
# Create group
security.groups.items = 1
security.groups.0.name = TestGroup
security.groups.0.description = This is a test group
security.groups.0.realm = myrealm
security.groups.0.authenticator = DefaultAuthenticator

################################
# Group membership
security.group.member.items = 1
security.group.member.0.user = TestUser
security.group.member.0.groups = TestGroup
security.group.member.0.realm = myrealm
security.group.member.0.authenticator = DefaultAuthenticator

3.8.2. Changing a user password

It is possible to change the password of an existing user.
To this end, add a security.users.0.update = true property:

sec-change-password.properties
script.log.level = DEBUG
script.log.file = changepwd.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Change a user password
security.users.items = 1
security.users.0.name = TestUser
security.users.0.update = true
security.users.0.password = welcome666
security.users.0.comment = This is an updated user
security.users.0.realm = myrealm
security.users.0.authenticator = DefaultAuthenticator

3.8.3. Policies

WLST-API lets you work with security policies.
You can create, update, read, import or list the currently configured policies.

Listing all policies

Working with Weblogic policies can sometimes be very cumbersome because the resource expressions are not documented.
Run the following properties file with the import script to list all policies registered in a Weblogic server:

sec-list-policies.properties
script.log.level = DEBUG
script.log.file = list_policies.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# List all policies
security.policies.items = 1
security.policies.0.mode = READ
security.policies.0.realm = myrealm
security.policies.0.authorizer = XACMLAuthorizer
Creating a policy

The following properties file create a policy to give access on the JMSModule to the group QueueBrowsers:

sec-create-policy.properties
script.log.level = DEBUG
script.log.file = create_policy.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Create policy on JMSModule
security.policies.items = 1
security.policies.0.mode = CREATE
security.policies.0.resource = type=<jms>, application=JMSModule
security.policies.0.expression = Grp(QueueBrowsers)
security.policies.0.realm = myrealm
security.policies.0.authorizer = XACMLAuthorizer
Reading a policy
security.policies.items = 1
security.policies.0.resource = type=<jms>, application=jmsModule2, destinationType=queue, resource=QUEUEIN2
security.policies.0.mode = READ
security.policies.0.realm = myrealm
security.policies.0.authorizer = XACMLAuthorizer
Updating a policy
security.policies.items = 1
security.policies.0.mode = UPDATE
security.policies.0.resource = type=<jms>, application=jmsModule2, destinationType=queue, resource=QUEUEIN2
security.policies.0.realm = myrealm
security.policies.0.authorizer = XACMLAuthorizer
security.policies.0.expression = Usr(myUser) | Usr(yourUser)
security.policies.0.expression.update.mode = REPLACE
Importing policies from a XACML file
security.policies.items = 1
security.policies.0.mode = BULK
security.policies.0.realm = myrealm
security.policies.0.authorizer = XACMLAuthorizer
security.policies.0.bulk.filepath = D:/app/WLSTAPI/export_security/policies_myrealm_XACMLAuthorizer.txt
security.policies.0.bulk.removeAll = false

3.8.4. Roles

WLST-API lets you create, update, read, or import roles.

Reading a role

Note that if no expression is assigned to the specified role, all roles will be displayed.

sec-read-role.properties
script.log.level = DEBUG
script.log.file = read_role.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Read role
security.roles.items = 1
security.roles.0.mode = READ
security.roles.0.name = Monitor
security.roles.0.realm = myrealm
security.roles.0.mapper = XACMLRoleMapper
Creating a role

The following will create a role called "MyOwnRole" that restricts queue access for QUEUEIN2 to users of group QueueUsers.

sec-create-role.properties
script.log.level = DEBUG
script.log.file = create_role.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Create role
security.roles.items = 1
security.roles.0.mode = CREATE
security.roles.0.name = MyOwnRole
security.roles.0.resource = type=<jms>, application=jmsModule2, destinationType=queue, resource=QUEUEIN2
security.roles.0.expression = Grp(QueueUsers)
security.roles.0.realm = myrealm
security.roles.0.mapper = XACMLRoleMapper
Updating a role
security.roles.items = 1
security.roles.0.name = monRole
security.roles.0.mode = UPDATE
security.roles.0.resource = type=<jms>, application=jmsModule2, destinationType=queue, resource=QUEUEIN2
security.roles.0.realm = myrealm
security.roles.0.mapper = XACMLRoleMapper
security.roles.0.expression = Usr(myUser)
security.roles.0.expression.update.mode = OR
Importing roles from a XACML file
security.roles.items = 1
security.roles.0.mode = BULK
security.roles.0.realm = myrealm
security.roles.0.mapper = XACMLRoleMapper
security.roles.0.bulk.filepath = D:/app/WLSTAPI/export_security/roles_myrealm_XACMLRoleMapper.txt
security.roles.0.bulk.removeAll = false

4. Removing resources

Removal of resource is provided by remove.py the script.

This scripts must be run against a running server.

The input properties file must contain the list of resources to be removed.
For each resource to be removed, you have to provide its name.

Check the Properties reference for a list of resources that can be removed (they all start with the prefix remove.).

4.1. Removing a JDBC data source

For instance, to remove a JDBC data source, you should provide a properties file similar to this one:

script.log.level = DEBUG
script.log.file = remove_jdbc.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Remove JDBC datasource
remove.jdbc.resource.list = MyDataSource

You can remove several data sources at once by separating their names with a comma.

This example is the most simple one as the data source does not rely on other resources.

4.2. Removing JMS resources

Other resources, such as JMS, may require more input to be provided.

For instance, if you want to remove a JMS queues as well as its module and server, you should provide an input that looks like this one:

script.log.level = DEBUG
script.log.file = remove_jdbc.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# Remove JMS resources
remove.jms.queue.items = 1
remove.jms.queue.0.module = JMSModule
remove.jms.queue.0.list = JMSQueue1

remove.jms.connectionfactory.items = 1
remove.jms.connectionfactory.0.module = JMSModule
remove.jms.connectionfactory.0.list = ConnectionFactory

# no need to remove subdeployment, it will be removed with the jms module
remove.jms.module.items = 1
remove.jms.module.0.list = JMSModule

remove.jms.server.list = JMSServer

In this case, the script will remove the queue JMSQueue1 (belonging to module JMSModule), the connection factory named ConnectionFactory, the JMS module itself as well as the JMS server (provided they exist on the target server).

5. Exporting resources

The export.py script handles the export of resources from a running Weblogic server.
The export is very handy if you don’t want to create the properties file (for the import) manually as it will generate it for you.
The export output can then be used by the import script to re-create the resources (supported by WLST-API).

The export, like the import and remove scripts, must be fed with a properties files indicating which resources you wish to export.

You specify the path of the exported propeties files with the export.output.filename property.

Once again, check the Properties reference for a list of resources that can be exported (they all start with the prefix export.).

5.1. Exporting JDBC resources

Would you want to export only JDBC resources, take a look at the following export defintion:

export-jdbc.properties
script.log.level = DEBUG
script.log.file = export_jdbc.log

################################
# Connection properties
domain.loading.type = connect
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

################################
# export properties

export.output.filename = Domain_export.properties

export.jdbc.datasource.enable = true
#export.jdbc.datasource.list=

This will export all JDBC resources defined in the target server.
As you can see, there is a commented property : export.jdbc.datasource.list
This one can be used to limit the export to the specifed JDBC data sources names (comma separated list).
Note that you can leave this property empty which has the same effect as if it was commented.

This principle applies for most of all the other exportable resources.

5.2. Exporting all resources (supported by WLST-API)
