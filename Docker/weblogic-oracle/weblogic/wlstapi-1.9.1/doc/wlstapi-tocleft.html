The official guide for the WLST-API tool. No more, no less.

Latest version is 1.9.1 (249), you can grab it here.

What’s new ?

1. Why WLST-API ?

Configurating a Weblogic domain can be a tedious and error prone process.

In the context of the European Commission, this process is even more risky because the teams reponsible for the deployment and configuration of Weblogic based applications (namely the Data Center) are different from those who develop them, the various Directorates General.

Legacy procedures consisted of providing a document explaining how to configure the domain.
This led to many human mistakes and a lot of resources wasted for both parties.

In order to improve this workflow, an automated tool was created on top of existing technical solutions: WLST-API.

So, currently, WLST-API is the link between development teams and the Data Center for all Weblogic domain configuration tasks.
Obviously, the tool can also be used to automate configuration tasks during the projet lifecycle.

2. What is WLST-API ?

WLST-API is a tool, built on top of WLST (WebLogic Scripting Tool), whose goal is to automate the creation and configuration of Oracle Weblogic domains.

It comes in the form of scripts, each having its own purpose.

Unlike WLST, the input for WLST-API are simple Java .properties files, thus relieving the burden of writing Jython scripts.

It is used by

  • Development teams (DG’s) to create resources on their DEV environments and automate some tasks during development.

  • Data Center to automate the creation of domains and configure them based on DG’s input (properties files).

WLST-API has been created and is maintained by the IPCIS Team at DIGIT-B.1.

In addition to this documentation, you can find a

  • Cookbook allowing you to quickly create a properties file for the required task

  • Properties reference documenting all the property names used by WLST-API

If you think this documentation can be improved, do not hesitate to contact us.

3. What WLST-API is not ?

WLST-API is not a replacement for WLST.
While it should in most cases address needs for the most used J2EE resources (JDBC, JMS), it does not implement the whole set of Weblogic resources. So, should you need an MBean/attribute not supported by WLST-API, you still have to resort to standard WLST jython script (sorry for that!).

WLST-API is not a deployment tool.

6. Running

  1. set WL_HOME environment variable (path to the Weblogic server home directory within your middleware installation, usually the wlserver or wlserver_10.3 directory)

    1. Unix systems (Linux, Solaris, Mac OS X)

      > export WL_HOME=/Users/Oracle/weblogic/wlserver

    2. Windows

      > set WL_HOME=C:\Data\Weblogic\wlserver

  2. run WLST-API

    1. Unix systems (Linux, Solaris, Mac OS X)

      > bin/wlstapi.sh SCRIPT SCRIPT_OPTIONS

    2. Windows

      > bin\wlstapi.cmd SCRIPT SCRIPT_OPTIONS

      • SCRIPT is the script to run, see here for the full list

      • SCRIPT_OPTIONS are the command line options and arguments for the script

Example: run an import:
	> bin/wlstapi.sh scripts/import.py --property import_jms_resources.properties
Always test your scripts (properties files) before submitting them to the Data Center.

6.1. JVM Options

If you need to pass JVM options (such as system properties for setting memory for instance), the easiest way is to set the JAVA_OPTION environment variable that will be picked by the start script.

Example: set JVM memory (Java < 1.8) for WLST-API (on Unix systems)
	export JAVA_OPTION="-Xms512m -Xmx512m -XX:PermSize=256m -XX:MaxPermSize=256m"
	bin/wlstapi.sh [arguments]

Please do not edit the start script as the next install will overwrite your changes.

7. How does it work ?

WLST-API requires two things in input:

  1. a script providing some functionality. Don’t panic, they are provided.

  2. options/arguments specific to each script; usually a properties file containing definition of resources, that you must write.

Now, we can have two types of scripts in WLST-API:

  • Offline scripts that can run without any server running

  • Online scripts that require a running server to connect to

See this link for more details on online/offline modes.

Offline scripts should be used mainly for domain creation. While previous versions of WLST-API allowed the creation of domain and resources in a single offline script, you should as much as possible separate the creation of the domain with a specific properties file (run with the domain creation script) and the creation of the resources with an import properties file (see below).

7.1. Scripts command line arguments

Even though each script may accept its own list of options and arguments from the command line, we have tried to offer a common and coherent interface. Check scripts details for possible variation from the standard options.

This is the usage for the online scripts:

Usage: path/to/script.py <options> --property <input.properties> [<input2.properties> ...]

options:
    -u,--user <username/password>
    -v,--version : display version
    -h,--help : display this help

The -u, --user option can be used to override authentication information that should otherwise be put in clear text in the properties file.

The offline scripts options differ only on the --user option: this option is obviously not available for offline scripts.

The attentive reader may have noticed that it is possible to pass in multiple properties files.

7.2. Script outcome

Upon invocation of WLST-API, the script may, as expected, run successfully or not.
We try as much as possible to catch runtime errors and exceptions and report them back but our scripts use WLST which is poorly designed in terms of exception management. Some WLST commands, when they fail, just print some output at the standard output or error stream and do not propagate the error up the calling stack.

Therefore, you should pay close attention to the script output and check if no error is reported (even if at the end, the script reports a successful run !).

In case of problem, a copy of the output (and the log file) should be sent along with any email to facilitate the investigation.

7.3.1. Domain creation

The domain creation script is essentially used by DC to create domains in their various environments (DEV, TEST, ACC, PROD) in an unattended way.

With this script, you can create from simple Weblogic domains to more complex ones (clustered domains).

The domain creation script uses templates to create domains.
Depending of the type of domain you want to create (Weblogic, OSB, SOA), you may have to add additional templates in the properties file to create the target domain.
Check the cookbook for examples.

7.3.2. Export

The export can be used to generate a properties file containing resources (supported by WLST-API) configured on the server.
Once exported, a properties file can be used as input for the Import script to re-create the resources on another server.

7.3.3. Import

This scripts allows the creation of resources.
So far, only the creation is possible, no update.

It works by reading the input properties file(s) and create the defined resources (JDBC, JMS, …​) in the server.
The order of creation does NOT depend on the order of properties in the file but is defined by the script itself, so you can arrange your properties file the way you like.

Any typo in a resource name will simply be ignored by the script. (A future version may add input properties validation).

The list of properties that can be used by this script can be found in the properties references document.
Check cookbook document for practical examples.

7.3.4. Remove

Allows the removal of resources.

7.4. Supported resources

While each script is independent, WLST-API tries to offer the same level of service (in terms of operations) for all supported resources. This means that for a given resource, it should be implemented for all operations: create (import.py), remove (remove.py) and export (export.py).

So far, the supported resources are:

  • Server, cluster and machine

  • Persistent store (file & jdbc)

  • JDBC data source

  • JMS server, module, connection factory, queue & topic, distributed queue & topic

  • SAF agent and resources (imported destination, SAF remote context, SAF error handling)

  • JMS Bridge

  • JMS Foreign server, destination and connection factory

  • Mail session

  • Work managers

  • Network channels

  • Startup/Shutdown classes

  • Security: user, groups, roles and policies

Note that only a subset of all available attributes are supported for each resource, the most common ones.
For the exhaustive list of supported resources and attributes, see the properties reference.

7.5. Properties files

Properties files are standard Java properties files, that is, text files that are human readable. See Properties class API for a complete reference of the format.

Properties files are composed of key-value pairs separated by a = character.
The order of the properties in the file is irrelevant, yet it makes sense to order it to ease readability.

Note: So far, no check is made on the property names, thus any invalid property name will simply be ignored by the script.
(note that we may add property validation in a future release)

For the complete list of accepted property names (keys), check the reference.

avoid the use of slash / or whitespace characters in the name of resources. You can use underscore, dash or dot instead. This limitiation applies only to resource names, not JNDI names.

The value of the properties depends on the property.
In any case, they are specified as text in the properties file.
The only thing to mention here is the boolean value: it can be either true or false.

7.5.1. Common properties

While the content of the properties file depends on the script, some properties are common to all WLST-API scripts:

Logging

Without any instruction, the WLST-API scripts do logging at the INFO level to the console.
You can change the level and output the log statements to a local file with the following properties:

Log control properties
script.log.level

either DEBUG, INFO, WARN or ERROR
script.log.file

a valid path to a local file (write access required)

Be aware that the log file will be overwritten on each subsequent run.

example
	script.log.level = DEBUG
	script.log.file = ${user.home}/logs/import_jms.log
	#...
Connection properties

The following properties are required for all online scripts:

Connection properties

  • domain.connect.url : URL of the server to connect to

  • domain.connect.username : username of the user connecting to the server

  • domain.connect.password : password of the user connecting to the server

the credentials (username and password) will be overridden by command line credentials -u, -user (if any).

7.5.2. Variable substitution

One convenient feature provided by the WLST-API is the ability to substitute variables in property values.
This is handy for instance when you have to repeat the same value for multiple properties (e.g. host name).
Actually, you can reuse every property value in another value using the ${property.name} syntax:

	#...
	mydb.jdbc.url = jdbc:oracle:thin:@server:1521:XE
	#...
	jdbc.datasource.0.driver.url = ${mydb.jdbc.url}

Even better, if the property file(s) loaded do not provide a value for a substitution variable, then the System properties will be queried.
This means you can use substitution variables such as ${user.home}, rather convenient indeed !

7.5.3. Indexed properties

To define a list of resources in WLST-API, you have to use indexed properties.
As the name suggests, these properties have an index in their name which determines the particular resource to which they relate.

For instance, if you want to create two data sources, you have to provide this:

	jdbc.datasource.items = 2 (1)

	# first JDBC definition
	jdbc.datasource.0.name = MyFirstDataSource
	jdbc.datasource.0.targets = AdminServer
	# (... rest of first data source definition)

	# second JDBC definition
	jdbc.datasource.1.name = MySecondDataSource
	jdbc.datasource.1.targets = AnotherServer
	# (... rest of second data source definition)
1 indexed property

The items property is merely used to depict the number of resources to define.
The index starts at zero.

This mechanism is used for all properties that describe list of resources.

7.5.4. Multiple properties files

As of version 1.7, it is possible to pass in multiple properties files in input. The goal of this feature is to allow to split a properties file in multiple ones. This can be very handy for instance when you want to define variables that depend on an environment. By using this feature, you only have to change the variables file while leaving the resources file unchanged.

In practice, assume you have this resources properties file:

jms.properties
domain.loading.type = connect
domain.name = MyDomain

target.server = ${jms.server.name}

# Persistent store
persistent.filestore.items = 1
persistent.filestore.0.name = FileStore
persistent.filestore.0.location = ${filestore.home}/jms_filestore
persistent.filestore.0.target = ${target.server}

# JMS Server
jms.server.items = 1
jms.server.0.name = JMSServer
jms.server.0.store = FileStore
jms.server.0.target = ${target.server}

# rest of JMS resources definitions omitted for brevity

You could then have the following properties file for variables in the DEV environment for instance:

vars-DEV.properties
domain.connect.url = t3://localhost:7001
domain.connect.username = weblogic
domain.connect.password = welcome1

jms.server.name = AdminServer
filestore.home = ${user.home}/temp/filestores

and this one for prod:

vars-PROD.properties
domain.connect.url = t3://prod-server.cec.eu.int:7453
domain.connect.username = weblogic
domain.connect.password = xxxxxxxx

jms.server.name = JmsServer
filestore.home = /path/to/filestores

Based on these input files, you would invoke WLST-API with this command line in DEV:

	> bin/wlstapi.sh scripts/import.py --property jms.properties vars-DEV.properties
The order of the properties file in the command line is important: a properties file overrides properties defined in the previous files (the last wins).

8. Limitations

So far, the update of existing resources (unless explicitly specified) is not supported. No support for this feature is foreseen in the near future. If you need to update a resource, first delete it then re-create it.

Another restriction is the the use of slashes / and whitespaces in resource names. We highly recommend NOT to use these characters. Note that this restriction applies only to resource names (e.g. JMS Queue name) not JNDI names.

11. Resorting to WLST

As mentioned above, if you need to automate a task that is not supported by WLST-API, you can still use WLST scripts. One way to do this is to use the Weblogic Admin Console recording capability which allows you to record actions on the domain and generate a WLST script.
Check WLST documentation for more information.
